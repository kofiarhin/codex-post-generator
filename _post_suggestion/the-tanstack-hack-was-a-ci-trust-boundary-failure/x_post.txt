The TanStack npm compromise was not just "npm got hacked."

It was a CI trust boundary failure with a much nastier lesson: if a poisoned cache can cross from a fork workflow into a release workflow, the attacker does not need your maintainer's laptop token.

They can wait for your own pipeline to mint a short-lived publish credential, then use it while everyone is still admiring the provenance badge.

OIDC trusted publishing is good engineering. So is provenance.

But neither fixes a workflow that lets untrusted code influence the release path.

The build pipeline is production now. Treat the cache, the workflow trigger, and the publish step like production attack surface.
