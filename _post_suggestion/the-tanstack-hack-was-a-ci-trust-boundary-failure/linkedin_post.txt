The TanStack npm compromise is the kind of supply chain incident that makes "trusted publishing" sound a little too optimistic.

Not because OIDC is bad. Not because provenance is useless. Not because maintainers were careless with a laptop token.

The uncomfortable part is better than that: the attacker reportedly used a CI path that looked legitimate enough to publish malicious package versions through the normal pipeline.

That is the part worth sitting with.

Modern software supply chain security has spent years moving away from long-lived secrets. Good. We should keep doing that.

But the TanStack incident shows the next layer of the problem: if GitHub Actions cache poisoning can bridge a fork workflow into a release workflow, then the publish credential does not need to be stolen from a human. It can be minted at the exact moment the attacker needs it.

OIDC trusted publishing raises the bar. It does not magically turn every workflow shape into a safe trust boundary.

The boring controls suddenly look less boring:

Audit `pull_request_target`.
Treat CI caches like executable infrastructure.
Pin third-party actions.
Monitor your own publishes.
Assume the build pipeline is part of the product, because attackers clearly do.

The lesson is not "never trust npm" or "open source is doomed."

It is simpler and more annoying: your release workflow is production.

And production deserves threat modeling before the incident report writes it for you.
