# Everyone Wants AI Coding Agents Until Security Joins the Standup

Everyone loves AI coding agents right up until they try to merge one into a real company.

In the demo, the agent writes tests, updates docs, fixes a flaky spec, and politely explains the diff like it has a computer science degree and perfect emotional regulation.

In production, it immediately meets six things no demo mentioned: permission boundaries, secret handling, weird internal APIs, compliance reviews, branch protection, and that one YAML file nobody understands but everyone fears.

That’s the funny part of the current AI tooling wave.

The biggest bottleneck is no longer “can the model write code?” It’s “can the workflow trust the model without creating a security incident, a legal headache, or a beautifully automated disaster?”

The teams getting real value are not treating agents like magic. They’re treating them like fast junior teammates with excellent pattern matching and zero institutional context.

That means scoped access, reviewable actions, narrow tool permissions, good audit trails, and very boring governance. Which is deeply unsexy and exactly why it works.

AI agents are not replacing engineering discipline.

They’re stress-testing whether you had any.

What’s been the bigger challenge for your team so far: model quality, or everything around the model?
